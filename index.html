php7升级之路

发表于 2016-08-31   |   分类于 PHP   |  

为了更高的性能，更低的服务器成本.
我们决定将接口业务的php从5.6升级到7.

升级过程

灰常简单

  • 安装

  • 编译依赖扩展
    注意有一些扩展可能不支持php7

  • 编辑php.ini
    注意开启opcache

PHP7的性能提升tips

  • opcache.file_cache 将opcode缓存到外部文件中,对一些脚本会有明显的性能提升.
  • gcc 4.8以上
    GCC 4.8版以上才會开启Global Register對opline和execute_data的支援，有助于提高5％的效能。
    接口业务经过性能测试后,发现并没有提升5%这么多.
  • 特定优化pgo, 可以对特定的程序进行优化.
    注意一定要进行性能测试, 我们的接口服务使用pgo后, 性能反而下降了.
  • Hugepage
    由于是虚拟机,而且内存不高,所以并没有开启
    如果系统有大量的物理内存（大于8G）,则可以开启,以提升性能.关于Hugepage,可以参考: Linux HugePage 特性

注意事项

  • 第三方扩展是否兼容php7
  • 移除了mysql_*相关函数
  • 一定要做完整的功能测试

迁移细则可见: 从PHP 5.6.x 移植到 PHP 7.0.x

坑?

PHP7没有坑,没有坑,没有坑

重要的事情说三遍

然而还有别的坑

在功能测试时, 发现某些接口会出现崩溃并重启进程的问题, php-fpm.log如下:

[25-Aug-2016 16:57:33] WARNING: [pool www] child 33056 said into stderr: “/tmp/php-7.0.9/Zend/zend_hash.c(1315) : ht=0x7fd187eb9c60 is being cleaned”

gdb调试之后也没有任何的头绪
随后设置断点，从扩展入手。
发现是一个ip扩展(monip)引起的问题,替换为其他方式获取ip后，问题得到解决.
所以一定要注意第三方扩展是否能在php7下正常工作.

PHP运行原理简介

发表于 2016-08-19   |   分类于 PHP   |  

作为一名PHP(拍黄片)程序员.
为了在提高自己对于PHP的理解, 让自己的专业技能(拍片)得到提高.
了解PHP的底层运行机制，是不可缺少的一环.

PHP的整体架构设计

通过该图来看, 从下往上, 我们把PHP解耦成几层.

  • Zend引擎
  • 扩展
  • php代码
  • SAPI

最上层为通过sapi进行交互的应用,比如apache,lighttpd,nginx.

Zend Engine

负责将代码解析为opcode,并执行.
这里的opcode指的是Zend虚拟机的指令,而不是cpu操作码.

Extensions

分为普通扩展(extension)和zend扩展(zend extension),通过扩展提供了许多内置工具.
比如array_walk等数组函数,对于redis,memcache的操作等.

PHP code

我们编写的PHP应用代码,大多数同学天天都在这一层写代码.

SAPI

（Server Application Programming Interface： the API used by PHP to interface with Web Servers）
PHP 通过SAPI 与外部应用进行交互
SAPI 隔离了外部环境和PHP, 并为内部的PHP提供了统一的接口,使得PHP不受外部影响
常见的有cli,cgi.

web环境下PHP的生命周期

以最常见的web环境举例, 通过一个HTTP请求来分析PHP的生命周期

php服务启动

作为服务初始化启动(php-fpm)

  • 初始化必要数据,常驻内存
  • 执行所有扩展的初始化函数MINIT
  • 等待请求

作为请求启动

  • PHP进程通过sapi获取到请求之后,开启一个新环境,初始化全局变量
  • 调用所有扩展的请求初始化函数RINIT

执行PHP代码

代码示例:

代码解析过程:

1.Scanning(Lexing) ,将PHP代码转换为语言片段(Tokens)

2.Parsing, 将Tokens转换成简单而有意义的表达式

3.Compilation, 将表达式编译成Opocdes

4.Execution, 顺次执行Opcodes，每次一条，从而实现PHP脚本的功能。

请求结束

  • 执行所有扩展的请求结束函数RSHUTDOWN
  • 释放变量表,内存
  • 将结果通过cgi sapi返回给nginx
  • nginx 返回 http reponse

服务结束

  • 执行所有扩展的初始化函数MSHUTDOWN
  • 释放内存

Git分支使用策略

发表于 2015-12-23   |   分类于 Git   |  

需求

  • 多个开发项目并行
  • 每个项目提交测试的代码,不可变更,上线也根据此代码包
  • 代码最终回归到主干,主干和线上代码一致
  • 每次上线,标记tag,备份留存
  • 快速拉取线上分支,修复bug

分支规范

分类

  • master 主干,永久且只有1个,一直都是最新的线上代码
  • develop-* 开发分支,多个,依照项目而定
  • release-* 发布分支,多个,来自develop,用于开发完成后的测试发布包,不允许修改
  • feature-* 功能分支,多个,来自develop,功能的拆解,各自开发人员的任务
  • fixbug-* 紧急修复bug分支,来自master,0个或1个

命名

  • develoop-* 星号为项目名称和加额外信息,附加信息可以是日期或是项目版本等.

  • release-* 星号为项目名称和提测版本,提测版本从1开始

  • feature-* 星号建议包含项目+姓名为前缀,因为是个人开发分支,自己分清即可

  • fixbug-* 星号为master主版本tag+fix版本号,fix版本号从1开始

master版本tag说明

  • 两位或三位版本号组成
  • 第一位代表重大版本更新
  • 第二位代表项目更新
  • 第三位如果存在,则代表bug修复

流程示例

开发流程

  • 从master拉取develop-*分支,此次我们拉取一个叫做develop-sample的分支
  • 在develop-sample进行开发,如有多个开发协作,可以从develop-sample拉取各自的feature分支
  • 开发完毕后,如有关联的feature分支,需要先合并feature分支,再删除feature分支.
  • 将develop-sample分支与master进行合并,再依据develop-sample拉取release-sample-1,将release-sample-1交由qa进行测试

bug修复开发流程

  • 从master拉取fixbug-*分支,此次我们拉取一个叫做fixbug-1.0.1的分支
  • 在fixbug-1.0.1进行开发
  • 将fixbug-1.0.1,发给qa进行测试
  • 测试无误后,将fixbug-1.0.1进行上线
  • 上线完毕后
    • fixbug-1.0.1合入到master
    • master标记tag 1.0.1
    • 删除fixbug-1.0.1
    • 若有release分支存活,则需要对应的develop分支重新合并master,重新发布release分支(主意更新版本号)

测试流程

  • 测试无误,执行上线流程
  • 测试有误,重新进行开发流程

上线流程

  • 将release-sample-1上线
  • 上线完毕后
    • release-sample-1合入到master
    • master标记版本tag 1.0
    • 删除release-sample-1分支
    • 删除branch-sample分支
